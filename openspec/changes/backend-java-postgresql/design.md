## Context

当前个人照片图库项目已基于 Node.js 和 SQLite 实现基本功能。为了增强后端稳定性、提高数据库扩展能力、并为后续运维和部署提供更稳健的基础，计划将后端实现迁移到 Java，并使用 PostgreSQL 作为后端数据库。

## Goals / Non-Goals

**Goals:**
- 使用 Java 重新实现后端 API 服务，保持现有功能和接口行为一致。
- 使用 PostgreSQL 作为持久化存储，支持更高的并发和更稳定的数据管理。
- 提供 Java 构建与运行配置，支持本地开发、测试和部署。

**Non-Goals:**
- 不引入多租户或团队共享功能。
- 不改变现有前端功能和用户体验。
- 不实现复杂的照片处理、AI 标签或外部分享功能。

## Decisions

- 采用 Spring Boot 作为 Java 后端框架。Spring Boot 提供快速构建、成熟生态和丰富的数据库集成支持，适合从 Node.js 迁移后的服务重构。
- PostgreSQL 用作关系型数据库。相比 SQLite，PostgreSQL 更适合生产环境、支持并发事务，并且可以容易迁移到云数据库服务。
- 维持现有 REST API 端点规范。前端与后端的交互方式保持不变，避免前端大量改动。
- 后端项目使用 Maven 或 Gradle 管理依赖和构建。为了兼容 Java 生态，优先选择常见的构建工具并提供启动脚本。
- 使用 JDBC / Spring Data JPA 访问 PostgreSQL。通过规范的数据模型和持久层实现，降低数据库迁移风险。

## Risks / Trade-offs

- [风险] Java 后端重写可能导致接口行为与现有 Node.js 版本出现差异。 → 通过保留现有 API 约定并编写兼容性测试来降低风险。
- [风险] PostgreSQL 引入配置和运维复杂性。 → 提供本地默认配置，并将数据库连接参数抽象到环境变量中。
- [风险] 迁移期间可能发生数据不一致或丢失。 → 先实现 schema 和接口，必要时再进行数据导入或同步工具支持。
