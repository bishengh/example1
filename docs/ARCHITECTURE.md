# 架构设计文档

## 系统概述

个人照片图库是一个面向单用户的轻量级 Web 应用，采用经典的三层架构：前端展示层、后端 API 层、数据存储层。系统设计遵循"简单可用"原则，避免过早引入微服务或复杂基础设施。

## 核心模块

### 1. 认证模块（`backend/src/routes/auth.js`）

基于 `express-session` 实现简单的 Session 认证。登录成功后，用户信息存入 Session Cookie；后续受保护接口通过 `requireLogin` 中间件校验 Session 有效性。

### 2. 照片管理模块（`backend/src/routes/photos.js`）

提供照片的 CRUD 操作：
- **上传**：使用 `multer` 接收 `multipart/form-data`，校验 MIME 类型后保存到本地文件系统
- **查询**：从 SQLite 读取照片元数据，拼接静态文件访问 URL 返回
- **更新**：支持修改标题、描述、标签和所属相册

### 3. 相册管理模块（`backend/src/routes/albums.js`）

提供相册的 CRUD 操作。删除相册时，通过 SQL `UPDATE` 将关联照片的 `album_id` 置为 `NULL`，保持数据一致性。

### 4. 前端模块（`frontend/`）

纯原生前端实现，无框架依赖：
- 使用原生 DOM 操作切换页面视图（登录页、图库页、上传页、相册管理页）
- 使用 `fetch` 与后端 API 通信
- 使用 CSS Grid 实现响应式照片卡片布局

## 技术决策

### 为什么选择 SQLite？

项目面向个人单用户场景，数据量可控，SQLite 无需额外部署数据库服务，零配置即可运行，符合"简单可用"的目标。

### 为什么选择本地文件系统存储照片？

为降低部署复杂度，当前版本使用本地磁盘存储上传文件，通过 Express 的 `express.static` 提供访问。后续若需扩展，可无缝替换为对象存储（如 S3/MinIO），只需修改 `storage.js` 模块的实现。

### 为什么前端不使用框架？

功能页面数量少（4 个主要视图）、交互复杂度低，使用原生技术栈可减少构建步骤和依赖体积，保持项目轻量。

## 扩展性说明

| 扩展方向 | 当前状态 | 建议方案 |
|----------|----------|----------|
| 多用户支持 | 仅支持单用户 | 扩展 `users` 表并在所有查询中加入 `user_id` 过滤 |
| 对象存储 | 本地文件系统 | 替换 `storage.js` 为 S3/MinIO 客户端 |
| 缩略图生成 | 未实现 | 上传后触发异步任务，使用 `sharp` 生成多尺寸缩略图 |
| 分布式部署 | 单机运行 | 将会话存储迁移到 Redis，数据库升级为 PostgreSQL |
